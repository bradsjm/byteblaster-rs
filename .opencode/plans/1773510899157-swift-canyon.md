# Incident Cleanup Scheduler Plan

## Summary
- Add a Postgres-specific cleanup API in `crates/emwin-db/src/postgres.rs` that expires incident rows still marked `active` when `end_utc` is in the past.
- Run cleanup once during `server` startup and then every 5 minutes from `emwin-cli` when Postgres persistence is enabled.
- Keep scheduler/process ownership in `emwin-cli`; keep mutation rules and SQL ownership in `emwin-db`.
- Do not change parser behavior, VTEC persistence rules, migrations, or CLI surface area for this iteration.

## Implementation Changes
- In `crates/emwin-db/src/postgres.rs`, add a small public result type such as `IncidentCleanupResult { expired_count: u64 }` and a method on `PostgresMetadataSink`:
  `pub async fn expire_active_incidents(&self, now: chrono::DateTime<chrono::Utc>) -> PersistResult<IncidentCleanupResult>`.
- Implement cleanup as a single indexed `UPDATE incidents` statement filtered by `current_status = 'active'`, `end_utc IS NOT NULL`, and `end_utc < $1`; set only `current_status = 'expired'` and `last_updated_at = now()`.
- Keep `latest_vtec_action`, `issued_at`, `first_product_id`, `latest_product_id`, and `latest_product_timestamp_utc` unchanged so synthetic cleanup does not masquerade as a real product update or interfere with stale-write protection.
- Re-export the cleanup result type from `crates/emwin-db/src/lib.rs` only if the method returns it publicly; otherwise keep all helper types crate-private.
- In `crates/emwin-cli/src/live/persistence.rs`, change Postgres startup wiring so server mode can retain a clone of the connected `PostgresMetadataSink` alongside the existing `PersistenceRuntime`; keep non-Postgres behavior unchanged.
- In `crates/emwin-cli/src/live/server.rs`, after Postgres sink connection succeeds and before the HTTP server starts serving, run one blocking cleanup pass with `Utc::now()`; if this startup pass fails, abort startup with a CLI runtime error.
- In `crates/emwin-cli/src/live/server.rs` (or a small helper in the live server module), spawn a Tokio cleanup task only when Postgres persistence is configured; the task should wait on a 300-second interval, call `expire_active_incidents(Utc::now())`, log nonzero expirations at info level, log failures at warn/error level, and continue running until `shutdown_rx` is signaled.
- Join the cleanup task during shutdown before the persistence runtime is torn down so the task cannot race a dropped pool/runtime.
- Do not add new CLI flags or env vars; hardcode the required cadence of startup plus every 5 minutes for this iteration.

## Test Plan
- Extend `crates/emwin-db/tests/postgres_sink.rs` with integration tests for:
  - active incidents with past `end_utc` becoming `expired`
  - active incidents with future `end_utc` remaining `active`
  - active incidents with `end_utc IS NULL` remaining unchanged
  - `cancelled`, `expired`, and `upgraded` incidents remaining unchanged
  - cleanup preserving `latest_vtec_action`, `latest_product_id`, `first_product_id`, and `latest_product_timestamp_utc`
- Add targeted unit coverage in `crates/emwin-db/src/postgres.rs` only if a helper/result mapping is introduced; otherwise prefer integration tests.
- Add `emwin-cli` unit coverage only if cleanup loop logic is factored into a testable helper; focus on startup/interval/shutdown behavior rather than duplicating DB assertions.
- Verify with:
  - `cargo fmt --all --check`
  - `cargo clippy --workspace --all-targets -- -D warnings`
  - `cargo test -p emwin-db --test postgres_sink`
  - `cargo test -p emwin-db`
  - `cargo test -p emwin-cli`
  - `cargo test --workspace`

## Assumptions
- Cleanup semantics are limited to synthetic expiration of rows currently marked `active`; it does not synthesize `EXP` as `latest_vtec_action` and does not alter lineage/timestamp guards used by real product upserts.
- The existing `incidents_current_status_end_utc_idx` added in `crates/emwin-db/migrations/0002_incidents.sql` is sufficient for this query; no migration change is needed.
- Sharing the existing single-connection Postgres pool is acceptable because the cleanup query is short and indexed; revisit only if ingestion latency shows contention.
- Startup cleanup failure should block server startup, while periodic cleanup failure should be logged and retried on the next interval.
